# Crypto Market Analysis Bot

Реалізовано: **всі 27 завдань ТЗ** — **1-11** (структура, головна, ринки, WebSocket, свічки, графік, індикатори, тренд, сигнали, сторінка аналізу, ризик), **12** (обране), **13** (авторизація), **14** (сповіщення), **15** (push), **16** (PWA), **17** (адаптація під iPhone), **18** (теми), **19** (база даних), **20** (історія сигналів), **21** (перевірка результатів), **22** (бек-тестування), **23** (обробка помилок), **24** (безпека), **25** (тести), **26** (розгортання — див. DEPLOYMENT.md), **27** (моніторинг). Біржа — **Bybit**.

## Структура

```
crypto-market-bot/
├── backend/     FastAPI, підключення до Binance REST API
└── frontend/    React + Vite + TypeScript + Tailwind
```

## Запуск backend

```bash
cd backend
python -m venv venv
source venv/bin/activate       
pip install -r requirements.txt
cp .env.example .env
uvicorn app.main:app --reload --port 8000
```

Перевірити:
- http://localhost:8000/health
- http://localhost:8000/api/markets
- http://localhost:8000/docs

## Запуск frontend

```bash
cd frontend
npm install
cp .env.example .env
npm run dev
```

Відкрити http://localhost:5173

## Що вже працює

- Відстежується **56 пар**: 15 великих монет (BTC, ETH, BNB, XRP, SOL, ADA, DOGE, TON, TRX, AVAX, LINK, DOT, LTC, BCH, ETC) + 41 менш капіталізована/нішева монета (ATOM, UNI, NEAR, APT, ARB, OP, SUI, INJ, мемкоїни PEPE/BONK/WIF/FLOKI/SHIB тощо). Повний список — `backend/app/services/exchange.py`.
- Backend тримає **одне спільне WebSocket-з'єднання** до Bybit (`wss://stream.bybit.com/v5/public/spot`) і роздає оновлення всім клієнтам через власні ендпоінти:
  - `WS /ws/markets` — усі пари одразу (головна сторінка)
  - `WS /ws/market/{symbol}` — одна пара (майбутня сторінка аналізу)
- При розриві з'єднання з біржею backend автоматично перепідключається (експоненційна затримка 1с → 30с).
- Frontend більше не робить polling — ціни оновлюються по WebSocket без перезавантаження сторінки, з таким самим автоперепідключенням і індикатором статусу.
- Список ринків має пошук по назві монети (55+ пар вже не влазять на екран без фільтра).
- Помилки біржі показуються користувачу без технічних деталей.
- `GET /api/candles/{symbol}?interval=15m&limit=500` — історичні OHLCV-свічки з Bybit. Підтримувані таймфрейми: `1m, 5m, 15m, 1h, 4h, 1d`. Приклад: http://localhost:8000/api/candles/BTCUSDT?interval=1h&limit=200
- Свічковий графік на TradingView Lightweight Charts під головною карткою — зелені/червоні свічки, обсяг, масштабування й переміщення (вбудовано в бібліотеку), кнопки таймфрейму `1m | 5m | 15m | 1h | 4h | 1d`.
- `GET /api/indicators/{symbol}?interval=1h` — технічні індикатори: SMA 20/50, EMA 20/50, RSI 14 (Wilder), MACD (12/26/9), Bollinger Bands (20, 2σ), ATR 14, зміна обсягу. Для кожного — поточне значення, попереднє і текстова інтерпретація. Модуль: `backend/app/analysis/indicators.py`.
- Тести математики індикаторів: `cd backend && pytest tests/ -v`. RSI звірено зі строгою Wilder-реалізацією (розбіжність < 0.000003 на 200 свічках).
- `GET /api/analyze/{symbol}?interval=1h` — повний аналіз: тренд (правила EMA з Завдання 8), сигнал Strong Buy/Buy/Neutral/Sell/Strong Sell за бальною системою з Завдання 9 (з поясненнями причин), рівні підтримки/опору через pivot-екстремуми, рівень ризику Low/Medium/High за нормалізованим ATR% (Завдання 11). Модулі: `backend/app/analysis/{trend,signals,risk}.py`.
- Тести тренду/сигналів/ризику: `tests/test_signals.py` — усі 11 меж бальної шкали, кожне бальне правило окремо, всі 4 кейси ризику.
- Сторінка аналізу `/analyze/{symbol}` (react-router): ціна, сигнал + бейдж ризику, смужка сили сигналу (score/8), тренд, список причин, попередження про волатильність, графік із синхронізованим таймфреймом (15m/1h/4h — МВП з ТЗ), плитки всіх індикаторів, підтримка/опір, час оновлення. Автооновлення кожну хвилину. На головній — кнопка "Full analysis", ринки клікабельні.
- Одне WebSocket-з'єднання на весь застосунок через `MarketStreamProvider` (React Context) — жодних дублікатів підключень.
- База даних (Завдання 19): усі 5 таблиць (Users, FavouriteAssets, Signals, Alerts, PushSubscriptions) на SQLAlchemy. Локально — SQLite (`backend/crypto_bot.db`, працює одразу, нічого встановлювати не треба); на проді достатньо задати `DATABASE_URL=postgresql+asyncpg://...` у `.env` і розкоментувати `asyncpg` у requirements.
- Фоновий рекордер сигналів (Завдання 20): кожні 5 хвилин аналізує BTC/ETH/SOL на 15m/1h/4h і зберігає запис, коли тип сигналу змінився (переходи, а не дублікати). Після рестарту відновлює останній стан із БД.
- `GET /api/signals` з фільтрами: symbol, signalType, timeframe, dateFrom/dateTo.
- Сторінка `/signals`: список сигналів з фільтрами (монета, тип, таймфрейм, дата), ціна на момент сигналу, сила, ризик, причини, результат через 1 годину (поки "pending" — заповнюватиметься в Завданні 21). Навігація Markets / Signals у шапці.

- Авторизація (Завдання 13): `POST /api/auth/register`, `POST /api/auth/login`, `GET /api/auth/me`. Паролі — тільки bcrypt-хеш із сіллю; JWT (HS256, 7 днів) підписується `JWT_SECRET` з `.env`; однакова помилка для неіснуючого email і неправильного пароля. Сторінки `/login`, `/register`, `/profile` з виходом. Тести: `tests/test_auth.py`.
- Обране (Завдання 12): зірочки ★ у списку ринків, фільтр "Favourites", керування порядком (↑/↓) і видалення в профілі. Для гостей — localStorage; після входу список об'єднується зі збереженим у БД і синхронізується через `GET/PUT /api/favourites` (додавання/видалення/порядок одним запитом).

- Сповіщення (Завдання 14): `POST/GET/DELETE /api/alerts`. Умови: ціна вище/нижче, RSI (1h) вище/нижче, сигнал (1h) став X — рівно приклади з ТЗ. Фоновий перевіряльник щохвилини: ціни з кешу WebSocket (безкоштовно), RSI/сигнал рахуються раз на пару. One-shot: після спрацювання видно час і фактичне значення. Архітектура нотифікаторів готова прийняти Web Push у Завданні 15. Сторінка `/alerts` з формою-реченням ("Notify me when BTC/USDT price rises above 70000"). Тести валідації: `tests/test_alerts.py`.

- PWA (Завдання 16): `manifest.json` (standalone, іконки 192/512/maskable, кольори теми), service worker з кешем оболонки (cache-first для статики, network-first для навігації, API не кешується), офлайн-фолбек, iOS-метатеги (`apple-touch-icon`, `apple-mobile-web-app-capable`). Іконки згенеровано у фірмовому стилі (`frontend/public/icons/`). Встановлення на iPhone: Safari -> Share -> Add to Home Screen. Для встановлення потрібен HTTPS (або localhost).
- Web Push (Завдання 15): `GET /api/push/public-key`, `POST/DELETE /api/push/subscribe`; надсилання через pywebpush + VAPID; прострочені підписки (410) видаляються автоматично. Push-нотифікатор підключений до alert_checker — спрацювання сповіщення доставляється на пристрій, клік відкриває `/analyze/{symbol}`. На сторінці Alerts — панель Enable/Disable з усіма станами (заборонено/не підтримується/не сконфігуровано). Налаштування: `python scripts/generate_vapid.py` -> ключі в `.env`. Обмеження iOS: push працює лише коли PWA додана на домашній екран (iOS 16.4+).

- Адаптація під iPhone (Завдання 17): нижнє меню Home / Markets / Signals / Alerts / Profile (фіксоване, з іконками, ховається на десктопі — там навігація в шапці), безпечні зони через `env(safe-area-inset-top/bottom)` (Dynamic Island + home-індикатор), `viewport-fit=cover`, `overflow-x: hidden` (без горизонтального прокручування), таби заввишки 56px (зручно великим пальцем), окрема сторінка `/markets` (дотик по монеті -> аналіз). На головній додано блок "Latest signals" (вимога Завдання 2).

- Теми (Завдання 18): вся палітра переведена на CSS-змінні (RGB-трійки — модифікатори прозорості Tailwind працюють). За замовчуванням — системна тема пристрою (стежимо за `prefers-color-scheme` наживо), перемикач у шапці циклом Auto -> Dark -> Light, вибір зберігається. Графік перефарбовується під тему, `theme-color` (смуга статусу iPhone) оновлюється. Заодно виправлено конфлікт токена `base` з Tailwind-класом розміру `text-base` (кнопки тепер використовують окремий токен `deep`).

- Перевірка результатів (Завдання 21): фоновий сервіс кожні 5 хв дозаповнює для кожного сигналу ціну через 15 хв / 1 год / 4 год (з історичних 5m-свічок) і макс. зростання/падіння за 4h-вікно. `GET /api/signals/stats` — точність: кількість оцінених, win rate за 1h (Buy — ціна зросла, Sell — впала; Neutral не оцінюється), середній результат у напрямку сигналу. На сторінці Signals — панель точності та колонки 15m/1h/4h + hi/lo для кожного сигналу. Міні-міграція в `init_db` додає нові колонки в наявну SQLite/PostgreSQL без ручних дій. Тести: `tests/test_outcomes.py`.

- Бек-тестування (Завдання 22): `GET /api/backtest/{symbol}?interval=1h&horizon=12` — прогін ТІЄЇ САМОЇ стратегії (той самий `generate_signal`) по 1000 історичних свічках; індикатори розраховуються векторно один раз, на кожному кроці використовуються лише дані до поточного моменту (без "підглядання в майбутнє"). Метрики: кількість сигналів, успішні, win rate, середній результат, макс. просадка (з компаундингом), найкраща/найгірша угода, profit factor, макс. серія збитків + останні 50 угод. Панель "Strategy backtest" на сторінці аналізу. Метрики звірені з ручним розрахунком, тести: `tests/test_backtest.py`.

- Обробка помилок (Завдання 23): глобальний обробник — технічні деталі лише в журнал, користувачу просте повідомлення; React ErrorBoundary (помилка сторінки не кладе застосунок); дружні помилки в кожному роутері; авто-перепідключення WS з обох боків; перевірка "недостатньо свічок".
- Безпека (Завдання 24): rate limiting у пам'яті (120/хв на API, 10/хв на auth проти перебору паролів, 6/хв на backtest), захисні заголовки (nosniff, X-Frame-Options DENY), bcrypt, JWT, секрети в .env, валідація всіх введень, API-ключі біржі взагалі не потрібні (публічні дані).
- Моніторинг (Завдання 27): журнал у консоль + `backend/app.log`; розширений `/health` (uptime, стан біржі/БД, кількість WS-клієнтів, кількість пар); автоперезапуск через healthCheckPath на Render або systemd `Restart=always` на VPS.
- Тести (Завдання 25): `cd backend && pytest tests/ -v` — індикатори (звірені з еталонним Wilder RSI), тренд, усі межі бальної шкали сигналів, ризик, авторизація (bcrypt/JWT), валідація сповіщень, результати сигналів, метрики бек-тесту, API-валідація і guard-и авторизації, захисні заголовки.
- Розгортання (Завдання 26): **див. [DEPLOYMENT.md](DEPLOYMENT.md)** — Render (backend, `render.yaml` готовий) + Vercel (frontend, `vercel.json` зі SPA-rewrite) + Neon (PostgreSQL).
